## 3. 性能瓶颈分析

### 3.1 串行执行

当前流程为**完全串行**：

```
文本审核 → 图片1提取 → 图片1审核 → 图片2提取 → 图片2审核 → ... → 图片5审核
```

每个步骤必须等待前一步完成，无法利用 I/O 等待时间。

### 3.2 重复/可合并操作

1. **图片文字提取**：每张图片单独调用 qwen-vl-max，共 5 次。qwen-vl-max 支持一次传入多张图片，可合并为 **1 次批量调用**。
2. **向量检索**：每次审核都单独检索，无法合并，但可与大模型调用并行（检索快，影响小）。

### 3.3 可并行点

| 可并行组 | 说明 | 预期节省 |
|----------|------|----------|
| 文本审核 + 批量图片提取 | 两者互不依赖，可同时进行 | 约 25 秒 → 取 max(4.6, 25) ≈ 25 秒（提取更慢） |
| 5 张图片的合规审核 | 提取完成后，5 个审核任务互不依赖 | 约 45 秒 → max(7.5, 13.5, 4.2, 9.5, 6.4) ≈ 13.5 秒 |

---

## 4. 优化方案

### 4.1 优化策略

1. **批量图片提取**：一次 qwen-vl-max 调用处理所有图片，减少 API 调用次数（5 次 → 1 次）。
2. **并行执行**：
   - 文本审核 与 批量图片提取 并行；
   - 5 张图片的合规审核并行。

### 4.2 预期性能

| 阶段 | 优化前 | 优化后 |
|------|--------|--------|
| 文本审核 | 4.6s | 4.6s（与提取并行，不增加总时间） |
| 图片提取 | 25.5s（5 次串行） | ~12s（1 次批量，通常略慢于单张） |
| 图片审核 | 45.7s（5 次串行） | ~14s（5 次并行，取最慢） |
| **总耗时** | **72.8s** | **约 18–22 秒** |

**预期提升**：从约 72 秒降至约 **18–22 秒**，缩短约 **70%**。

---

## 5. 实现要点

- 使用 `concurrent.futures.ThreadPoolExecutor` 实现并行（兼容同步 `review` 接口）。
- 新增 `_parse_batch_extraction_output` 解析 `【图片1】【图片2】...` 格式，得到每张图片的文本。
- 保持返回结构不变，确保与现有前端和 API 兼容。
- 解析失败时，将整段文本作为第一张图片内容兜底。
